## Executive Summary

### Project Vision

**骑行即征服——每一公里都在改变全国领土版图。**

顽鹿骑行征服是一款嵌入顽鹿运动App的H5策略游戏，将真实骑行里程转化为全国2800个县区级领土的攻防力量。玩家以战队为单位攻城掠地，通过异步协作（无需同时骑行）和周循环竞争（每周一轮攻防），让每一次独自骑行都服务于集体征服目标。

核心体验承诺：
- **情感先于规则**：用户首次打开看到的不是游戏规则，而是"你的家乡正在被守护"或"你的家乡还是无主之地"
- **每骑必反馈**：骑行结束立即看到"你的23km推动了攻城进度从52%到58%"
- **零管理协作**：会长操作<30秒（智能推荐+一键确认+不操作自动兜底）
- **集体叙事**：战队的领土扩张自然生成史诗故事——"XX铁骑统一山东省"

### Target Users

**Alpha阶段核心用户（4种角色）：**

| 用户类型 | 代表人物 | 核心需求 | 使用频率 |
|---|---|---|---|
| **战队骑友** | 老周，35岁，每周骑3-4次 | 骑行有集体意义、看到贡献影响、竞争乐趣 | 每天（骑行后+非骑行回访） |
| **战队会长** | 张哥，42岁，战队30人 | 零管理成本、战队自运转、成就感 | 每周1-2次（确认作战计划+查看战报） |
| **新用户** | 小王，24岁，骑行半年 | 30秒内理解核心概念、产生情感连接 | 首次体验决定留存 |
| **运营管理员** | 小刘，运营团队 | 数据监控、异常处理、结算验证 | 每日（Alpha通过脚本） |

用户的共同特征：移动端使用、竖屏/横屏双模式、骑行爱好者、习惯在顽鹿运动App内活动。技术素养中等偏上（会使用运动类App），但不期望学习复杂游戏规则。

### Key Design Challenges

**1. 地图信息密度管理**
2800个县区在竖屏/横屏双模式下的展示是核心挑战。需要在"全国宏观感知"和"单区域详细信息"之间找到平衡。三级下钻（全国→省→县区）必须直觉化操作，手势缩放和点击下钻的交互模式需要零学习成本。

**2. 周循环状态感知**
游戏有4个阶段（对战期→结算期→战报发布→宣战窗口），用户需要随时知道"现在是什么阶段、还剩多少时间、我该做什么"。UI必须在不占用太多空间的前提下持续传达这个信息。

**3. 30秒首次体验的"魔法时刻"**
PRD Journey 4明确要求：新用户30秒内从"这是什么？"到"我的骑行跟家乡有关系了"。这要求首屏加载极快（骨架屏<500ms）、自动定位到家乡、情感文案优先于规则说明、首次交互极简（一个按钮完成"贡献里程"动作）。

**4. HTTP轮询下的"实时感"营造**
架构决策为30秒HTTP轮询（非WebSocket），但攻防进度需要给用户"实时推进"的感觉。前端需要用动画过渡、进度条渐变等视觉手法弥补数据更新的间隔感。

### Design Opportunities

**1. 骨架屏即叙事起点**
三层渲染策略（骨架屏→简化地图→完整数据）不仅是技术优化，更是叙事机会。骨架屏可以是深色系地图轮廓逐渐亮起的动画——用户等待加载的0.5-1.5秒内，已经开始感受"科技地理"的视觉氛围。

**2. 周循环的仪式感时刻**
每周日12:00战报发布、每周一12:00宣战公布——这些固定时刻是天然的仪式感设计机会。战报不只是数据陈列，而是一个"本周战争故事"的叙事体验；宣战公布可以设计成"战鼓擂响"式的全屏动效。

**3. 地图即社交画布**
地图不仅展示领土归属，更是社交互动的承载面。用户通过地图发现"隔壁城市有个很强的战队"、"我的家乡正在被攻打"——地图上的每一次变化都是社交话题的种子。

## Core User Experience

### Defining Experience

**核心用户动作：骑行后打开游戏，看到自己的里程推动了攻城进度变化。**

这是整个产品的价值锚点——不是地图浏览、不是战队管理、不是排行榜，而是"我骑了一圈回来，世界因此不同了"。

**核心体验循环：**

```
真实骑行 → 打开H5 → 看到进度变化（"你的23km: 52%→58%"）→ 情感满足 → 浏览地图/战队 → 期待下次骑行
```

整个UX设计围绕这个循环展开：进入即反馈、反馈即情感、情感即留存。

### Platform Strategy

**平台：横屏 H5，运行在顽鹿运动 App WebView 内**

**核心定位**：横屏游戏化布局，营造「战略指挥屏」沉浸体验。

| 约束条件 | 具体参数 | UX 影响 |
|---|---|---|
| 屏幕尺寸 | 1920px × 1080px 横屏（基准） | 地图占据 85% 屏幕，沉浸式视觉 |
| 运行环境 | App 内 WebView（非独立浏览器） | 无浏览器导航栏、Hash 路由 |
| 页面结构 | 5 个核心页面 | 底部功能栏 + 顶部信息条，横屏布局 |
| 首屏性能 | 骨架屏<500ms，可交互<3s | 三层渐进渲染策略 |
| 包体积 | 首包<200KB gzipped | 精简依赖，按需加载 |
| 数据更新 | HTTP 轮询 30 秒间隔 | 前端动画弥补非实时感 |
| 触控优先 | 手指/鼠标双模式 | 点击目标≥48px，手势缩放 + 点击下钻 |
| 响应式断点 | ≥1920px 桌面 / 1024-1920px 平板 / 769-1024px 手机横屏 | 等比缩放 + 功能栏简化 |

### Effortless Interactions

**零思考交互（用户不需要任何思考）：**

| 交互 | 实现方式 | 用户感知 |
|---|---|---|
| 骑行后看到贡献 | 打开即显示本次骑行贡献浮层 | "我的努力被看到了" |
| 知道当前阶段 | 顶部常驻周循环状态条 | "现在是对战期，还剩3天" |
| 发现家乡位置 | 首次自动定位+高亮 | "我的家乡在这里" |
| 了解战队状态 | 战队Tab一屏展示核心数据 | "我们队目前占了12个县" |
| 查看攻防进度 | 点击领土即显示进度详情 | "这个县快要打下来了" |

**最少步骤交互（1-2步完成）：**

| 交互 | 步骤数 | 流程 |
|---|---|---|
| 会长确认作战计划 | 1步 | 系统智能推荐→一键确认 |
| 查看周战报 | 1步 | 战报Tab→全屏战报卡片 |
| 切换地图层级 | 1步 | 双指缩放 或 点击省份下钻 |
| 加入战队 | 2步 | 接受邀请链接→确认加入 |
| 新用户首次体验 | 2步 | 自动定位家乡→点击"开始征服" |

### Critical Success Moments

**按影响力排序的关键时刻：**

**🔴 时刻1：首次打开的前5秒（影响力：60%用户留存）**
- 用户必须在5秒内看到与自己有关的内容（家乡位置、骑行数据）
- 失败模式：空白加载、规则说明、注册表单
- 设计要求：骨架屏→地图→自动定位家乡，零操作到达情感触点

**🔴 时刻2：首次骑行后的反馈（影响力：80%二次打开率）**
- 用户骑行后打开H5，必须立即看到"你的里程产生了什么影响"
- 失败模式：需要手动查找、数字变化不明显、无情感文案
- 设计要求：浮层自动展示贡献、进度条动画推进、情感化文案（"你的23km让战队更近一步"）

**🟡 时刻3：首次周战报（影响力：周留存转化）**
- 用户第一次看到完整周战报，必须感受到"这一周的骑行有了集体成果"
- 设计要求：叙事化战报（不是数据表格）、战队荣誉高亮、个人贡献占比可视化

**🟡 时刻4：会长首次操作作战计划（影响力：战队活跃度）**
- 会长打开作战计划，必须在30秒内完成确认
- 设计要求：智能推荐前置、一键确认、不操作12小时自动执行兜底

### Experience Principles

**UX决策优先级框架（从高到低）：**

| 优先级 | 原则 | 含义 | 冲突时让步方 |
|---|---|---|---|
| P0 | **反馈优先 (Feedback First)** | 任何用户动作都必须有即时视觉反馈 | 其他所有原则 |
| P1 | **情感先于逻辑 (Emotion Before Logic)** | 先触动用户情感，再解释规则 | 信息完整性 |
| P2 | **一目了然+一键触达 (Glanceable & One-Tap)** | 核心信息扫一眼就懂，核心操作点一下就行 | 功能丰富度 |
| P3 | **智能默认 (Smart Defaults)** | 系统替用户做决定，用户只需确认 | 自定义灵活性 |
| P4 | **地图即灵魂 (Map as Soul)** | 地图是核心视觉和交互载体，不是附属元素 | 页面布局对称性 |

**原则应用示例：**
- 当"信息完整"和"情感触动"冲突时 → 先展示情感文案，详细数据放在二级展开
- 当"功能丰富"和"一键触达"冲突时 → 砍功能，保简洁
- 当"自定义选项"和"智能默认"冲突时 → 提供默认值，高级选项折叠隐藏

## Desired Emotional Response

### Primary Emotional Goals

**产品的情感核心：让骑行从"孤独的运动"变成"集体征服的史诗"。**

| 情感目标 | 触发场景 | 设计表达 |
|---|---|---|
| **归属感 (Belonging)** | 看到战队领土、成员骑行贡献 | 地图上战队颜色覆盖、"你的战队占领了XX"文案 |
| **成就感 (Accomplishment)** | 骑行后看到进度推进、攻下新领土 | 进度条动画、领土攻克全屏庆祝动效 |
| **被看见 (Recognition)** | 个人里程贡献被量化展示 | "你的23km贡献了38%的攻击力"浮层 |
| **期待感 (Anticipation)** | 周循环倒计时、宣战公布 | 倒计时脉冲、"战鼓擂响"式宣战动效 |
| **掌控感 (Agency)** | 会长制定作战计划、骑友选择贡献方向 | 地图点击即攻击、一键确认计划 |

**一句话情感承诺：** 每次打开，都能感受到"我的骑行正在改变这张地图"。

### Emotional Journey Mapping

**用户情感旅程（按关键时刻）：**

```
首次打开         骑行后回访         日常浏览         周战报         宣战公布
   │                │                │              │              │
   ▼                ▼                ▼              ▼              ▼
 好奇→惊喜       期待→满足        安心→自豪      震撼→荣耀      紧张→兴奋
"我的家乡！"   "我推动了进度"   "我们越来越强"  "这周真牛"    "下周打哪里"
```

**各阶段情感详解：**

| 阶段 | 目标情感 | 反面情感（必须避免） | 设计保障 |
|---|---|---|---|
| **首次打开（0-5秒）** | 好奇 → 惊喜（"这跟我有关"） | 困惑、无聊、被要求学习 | 自动定位家乡、零文字教程、情感文案优先 |
| **首次骑行后** | 期待 → 满足（"我的努力有效果"） | 失望（找不到变化）、被忽视 | 自动弹出贡献浮层、进度动画、个人数据高亮 |
| **日常地图浏览** | 安心 → 自豪（"我们在扩张"） | 焦虑（太复杂）、迷失（找不到重点） | 战队领土颜色醒目、一眼看到占领数 |
| **周战报发布** | 震撼 → 荣耀（"这一周值了"） | 无感（只是数字）、挫败（全输了） | 叙事化战报、即使输也高亮积极数据 |
| **宣战公布** | 紧张 → 兴奋（"新的一周开始了"） | 无所谓（跟我无关） | 全屏宣战动效、明确"你的战队将进攻XX" |
| **会长操作** | 掌控 → 轻松（"30秒搞定"） | 压力（太多选择）、愧疚（没时间管理） | 智能推荐+一键确认、不操作自动兜底 |
| **出错/加载慢** | 耐心 → 信任（"它在加载中"） | 烦躁、怀疑是否卡死 | 骨架屏渐进动画、明确的加载状态 |

### Micro-Emotions

**关键微情感对及设计策略：**

**✅ 必须强化的正向微情感：**

| 微情感 | 触发时机 | 强化手段 |
|---|---|---|
| **"我很重要"** | 看到个人里程贡献占比 | 精确到km的个人贡献数据，不是模糊的"参与了" |
| **"我们很强"** | 浏览战队领土版图 | 战队颜色在地图上的视觉面积，领土数量醒目展示 |
| **"还差一点点"** | 攻城进度接近100% | 进度条最后10%放大显示，脉冲动画暗示"就差你这一骑" |
| **"不骑就亏了"** | 周循环倒计时 | 柔和的紧迫感文案（"距离本周结算还有2天"），不是焦虑式催促 |
| **"意料之外"** | 领土被偷袭、意外攻克 | 地图变化推送通知风格的浮层，制造惊喜叙事 |

**❌ 必须消除的负向微情感：**

| 负向微情感 | 可能触发场景 | 消除策略 |
|---|---|---|
| **"我不懂规则"** | 首次接触游戏机制 | 不解释规则，用结果说话（"你骑了→进度涨了"） |
| **"我的贡献没用"** | 攻城失败、领土被夺 | 即使失败也展示"你的贡献防守了XX%"，永不归零 |
| **"太复杂了"** | 地图信息过多 | 默认简化视图，详情按需展开 |
| **"又要管理"** | 会长收到操作提醒 | 智能默认值让"不操作"也是合理选择 |
| **"没人在意"** | 骑行后无反馈 | 每次骑行后必弹贡献浮层，即使贡献极小也正面表达 |

### Design Implications

**情感 → 设计决策映射表：**

| 情感目标 | 视觉设计 | 交互设计 | 文案策略 |
|---|---|---|---|
| **归属感** | 战队专属颜色贯穿全局、领土着色饱和度表达占领强度 | 首次自动定位家乡省份 | "你的家乡"、"你的战队"——用"你的"建立所有权感 |
| **成就感** | 进度条从暗到亮的渐变、攻克时领土颜色脉冲扩散 | 攻克瞬间全屏庆祝动效（轻量，<1秒） | "攻克！"、"XX铁骑拿下了浦东新区" |
| **被看见** | 个人贡献数据用高亮色独立展示 | 贡献浮层自动弹出，不需要用户查找 | 精确里程数+"你的XX km推动了..." |
| **期待感** | 倒计时用脉冲呼吸灯效果 | 周循环状态条常驻可见 | "距离宣战还有..."、"新的战场即将揭晓" |
| **掌控感** | 可操作元素明确的视觉可供性（按钮形态清晰） | 一键操作、智能推荐前置 | "确认"而非"选择"——系统已经帮你想好了 |

**情感安全网（防止负面情感）：**

| 风险场景 | 情感风险 | 兜底设计 |
|---|---|---|
| 全部领土被夺 | 挫败、放弃 | "失地可以夺回"强调可逆性、展示防守贡献 |
| 骑行量极少（1km） | 自己没用 | "每一公里都计入攻击力"、不设最低门槛 |
| 战队无人骑行 | 孤独感 | 展示全国战况，即使自己战队静默也能看到"世界在动" |
| 长时间未打开 | 脱节感 | 回访时摘要浮层："你离开期间，发生了这些变化..." |

### Emotional Design Principles

**情感设计五条铁律：**

| # | 铁律 | 具体含义 | 违反案例 |
|---|---|---|---|
| 1 | **永远正面表达** | 任何数据都用正向框架呈现，即使是失败 | ❌ "你的领土被夺了3个" → ✅ "你成功防守了8个领土" |
| 2 | **每骑必响** | 任何骑行（哪怕1km）后打开都有专属反馈 | ❌ 骑行后打开无任何变化 |
| 3 | **紧迫不焦虑** | 周循环倒计时营造期待感，不制造FOMO | ❌ "你还没骑！再不骑就来不及了" → ✅ "本周对战还剩3天，你的每一骑都算数" |
| 4 | **集体叙事优先** | 个人数据永远放在集体故事的语境中 | ❌ "你骑了23km" → ✅ "你的23km助力战队攻城进度+6%" |
| 5 | **失败也有收获** | 输掉对战也要让用户感觉这一周有价值 | ❌ "本周失败" → ✅ "本周虽然失守2城，但你的战队总骑行量增长了40%" |

## UX Pattern Analysis & Inspiration

### Inspiring Products Analysis

**产品1：Strava — 运动社交的标杆**

| 维度 | 分析 |
|---|---|
| **核心UX优势** | 运动结束后的即时反馈循环：完成骑行→自动上传→看到成绩+排名+点赞 |
| **入门体验** | 极简——注册后直接骑，第一次骑行后才展开社交功能 |
| **信息层级** | Feed流为主，"我的成绩"优先于"社交内容"，个人成就感驱动 |
| **创新交互** | Segment（路段）排行榜——把普通道路变成赛场，骑过的路自动参与排名 |
| **留存机制** | "连续骑行天数"、月度总结、年度回顾——数据积累产生沉没成本 |
| **对我们的启示** | "每骑必反馈"的核心体验循环与Strava一致，但我们的反馈是"攻城进度"而非"个人成绩" |

**产品2：Clash of Clans（部落冲突）— 异步团队策略的典范**

| 维度 | 分析 |
|---|---|
| **核心UX优势** | 地图即游戏——所有操作发生在一张可视化的基地地图上 |
| **入门体验** | 引导式教程：3步内完成第一次建造，立即看到基地变化 |
| **信息层级** | 基地全景→点击建筑看详情→操作面板，三级下钻与我们的全国→省→县一致 |
| **创新交互** | 部落战的异步协作——成员不需要同时在线，各自完成攻击即可 |
| **留存机制** | 建造倒计时+部落战周期——定期回访检查进度 |
| **对我们的启示** | 异步协作模式（各自骑行，贡献汇总）直接可迁移；"地图即核心交互面"的理念与我们的P4原则吻合 |

**产品3：微信读书 — 轻量社交+进度可视化**

| 维度 | 分析 |
|---|---|
| **核心UX优势** | 把"孤独的阅读"变成"有同伴的阅读"——阅读时长排行、好友在读、想法互动 |
| **入门体验** | 无需教程，打开即书架，核心动作（阅读）零门槛 |
| **信息层级** | 极简首页——书架+排行+发现，三个Tab覆盖全部功能 |
| **创新交互** | 阅读时长自动转化为"书币"——行为即奖励，与我们"骑行即攻城"的理念高度相似 |
| **留存机制** | 每周阅读排行+阅读时长统计——社交压力+数据积累双重驱动 |
| **对我们的启示** | "行为自动转化为游戏资源"的核心模式完全一致；周排行榜机制可参考其社交展示方式 |

### Transferable UX Patterns

**导航模式：**

| 可迁移模式 | 来源 | 应用场景 |
|---|---|---|
| **底部功能栏（4个按钮+头像）** | Strava、微信读书 | 5个核心页面：地图/战队/作战/战报/排行榜 |
| **地图三级下钻（缩放+点击）** | 部落冲突、高德地图 | 全国→省→县区的层级浏览 |
| **浮层式详情（点击弹出，不跳页）** | Strava路段详情 | 点击县区显示攻防详情，不离开地图页 |

**交互模式：**

| 可迁移模式 | 来源 | 应用场景 |
|---|---|---|
| **行为自动转化为游戏资源** | 微信读书（阅读→书币） | 骑行里程→攻击力，零操作完成 |
| **异步贡献汇总** | 部落冲突（各自攻击→汇总战果） | 战队成员各自骑行→攻城进度汇总 |

**视觉模式：**

| 可迁移模式 | 来源 | 应用场景 |
|---|---|---|

### Anti-Patterns to Avoid

| 反模式 | 案例 | 为什么要避免 | 我们的替代方案 |
|---|---|---|---|
| **强制教程** | 早期手游的"点击这里→点击那里"引导 | 用户耐心极低，30秒内流失 | 零教程，用结果说话（自动定位+情感文案） |
| **数据过载首页** | 某些运动App首页塞满图表 | 违反P2原则（一目了然） | 首页=地图，数据按需展开 |
| **惩罚性通知** | "你已经3天没骑了！" | 违反铁律3（紧迫不焦虑） | 正向框架："本周对战还在继续，你的每一骑都算数" |
| **复杂设置流程** | 需要填写大量信息才能开始 | 违反P3原则（智能默认） | Alpha阶段用户数据来自顽鹿App，零设置 |
| **胜负零和展示** | "你输了"全屏红色提示 | 违反铁律1（永远正面表达）和铁律5（失败也有收获） | 即使输也高亮正面数据 |
| **功能入口堆砌** | 首页放满功能入口图标 | 违反P4原则（地图即灵魂） | 地图是首页唯一主角，其他功能通过Tab切换 |

## Design System Foundation

### Design System Choice

**方案：Tailwind CSS + 项目专属轻量组件库**

不采用现成UI框架（如Ant Design Vue、Element Plus），而是基于Tailwind CSS的原子化样式能力，构建项目专属的轻量组件库。

| 决策维度 | 选择 | 备选（已排除） |
|---|---|---|
| CSS框架 | **Tailwind CSS** | 纯CSS / Sass（开发效率低） |
| UI组件库 | **项目自建轻量组件** | Ant Design Vue（太重，~300KB+）|
| 图标方案 | **SVG内联图标（精选集）** | Icon Font（不够灵活） |
| 地图渲染 | **ECharts Map（按需加载）** | D3.js（学习成本高、包体大） |
| 动画方案 | **CSS Transitions + Vue Transition** | GSAP（额外依赖，增加体积） |

### Rationale for Selection

**为什么选Tailwind CSS + 自建组件：**

| 考量因素 | 分析 | 结论 |
|---|---|---|
| **包体积** | 首包<200KB硬约束；Ant Design Vue仅CSS就100KB+ | Tailwind经purge后通常<10KB |
| **视觉独特性** | 「科技地理」深色风格、脉冲光效、领土着色——标准UI库无法提供 | 必须自定义视觉层 |
| **组件需求** | 5个页面，核心交互集中在地图；传统UI组件（表格/表单/菜单）需求极少 | 不需要完整UI库 |
| **开发效率** | Tailwind的原子化类名+Vue 3组合式API = 快速构建自定义组件 | 开发效率不亚于UI框架 |
| **维护成本** | 自建组件数量少（预计15-20个），维护负担可控 | 长期可维护 |
| **团队适配** | 架构已明确Vue 3 + TypeScript技术栈 | Tailwind与Vue 3集成成熟 |

**为什么不选现成UI框架：**

| 框架 | 排除原因 |
|---|---|
| Ant Design Vue | 体积过大（200KB+）；组件风格偏企业后台，不适合游戏化H5 |
| Element Plus | 同上；且视觉风格偏传统 |
| Vant（移动端） | 偏电商/工具类App；地图和游戏化组件完全缺失 |
| Naive UI | 相对轻量但仍有60KB+；仍需大量定制 |

## Core Interaction Design

### Defining Experience

**一句话定义：骑车攻城——每一公里骑行自动变成地图上看得见的领土变化。**

**经典产品类比：**

| 产品 | Defining Experience | 我们的版本 |
|---|---|---|
| Tinder | 滑动匹配人 | — |
| Strava | 骑完看数据和排名 | 骑完看攻城进度 |
| 部落冲突 | 建造和攻打基地 | 骑行攻打真实县区 |
| **顽鹿骑行征服** | **骑车→地图变色** | 真实骑行→真实地图上的领土颜色变化 |

**核心交互的独特价值：**
- 不需要用户"玩游戏"——骑行本身就是游戏动作
- 不需要学规则——进度条涨了就是在赢
- 不需要管理——系统自动把里程转化为攻击力
- **骑行行为和游戏结果之间的因果关系一目了然**

### User Mental Model

**用户带着什么心智模型来：**

| 心智模型来源 | 用户预期 | 我们如何对齐 |
|---|---|---|
| **运动App（Strava/Keep）** | 骑完看数据、看成绩 | ✅ 骑完看贡献数据，但框架是"攻城"不是"成绩" |
| **战争策略游戏** | 点击操作、主动进攻 | ⚠️ 需要重塑：攻击不是点击，是骑行；打开App是看结果，不是操作 |
| **微信运动/步数排行** | 运动量自动转化为社交比较 | ✅ 完全一致：骑行量自动转化为战队竞争力 |
| **地图类App（高德/百度）** | 捏合缩放、点击看详情 | ✅ 地图交互模式完全复用 |

**关键心智转换（必须在5秒内完成）：**

```
用户打开前的心智：    "这是一个骑行App的功能"
      ↓ (0-2秒)       看到深色地图 + 领土着色
用户5秒后的心智：     "这是一张我的战队正在征服的地图"
```

**可能的困惑点及预防：**

| 困惑 | 原因 | 预防设计 |
|---|---|---|
| "我需要做什么？" | 不知道游戏目标 | 首屏自动定位家乡+情感文案："你的家乡正在被守护" |
| "骑行怎么变成攻击？" | 不理解转化机制 | 不解释机制，直接展示结果："你的23km → 进度+6%" |
| "我怎么选择攻打哪里？" | 误以为需要主动操作 | 明确提示"会长已选定目标"或"系统自动分配" |
| "我的骑行算了吗？" | 担心数据未同步 | 骑行后第一屏必显贡献浮层，消除疑虑 |

### Success Criteria

**核心交互成功的量化标准：**

| 标准 | 指标 | 判定方法 |
|---|---|---|
| **即时可理解** | 新用户5秒内理解"骑行=攻城" | 首次打开无需点击任何说明按钮 |
| **因果可见** | 骑行后2秒内看到贡献数据 | 贡献浮层自动弹出，进度条动画启动 |
| **情感共鸣** | 用户感到"我的骑行有意义" | 文案使用"你的XX km"个人化表达 |
| **零操作攻城** | 骑行→攻城全自动，无需手动操作 | 用户打开只是"看结果"，不是"做操作" |
| **回访欲望** | 用户想知道"现在进度怎样了" | 地图变化持续发生（30秒轮询），每次打开都有新信息 |

**"This Just Works" 清单：**

- ✅ 打开就看到地图（不是加载页、不是教程）
- ✅ 地图上有我的战队颜色（不是空白地图）
- ✅ 骑行后有我的贡献数据（不需要翻找）
- ✅ 进度在变化（不是静态截图）
- ✅ 知道现在是什么阶段（不需要猜）

### Novel UX Patterns

**模式分类：我们的交互是"熟悉模式的创新组合"**

| 交互元素 | 模式类型 | 说明 |
|---|---|---|
| 地图缩放浏览 | ✅ 已有模式 | 复用地图App的捏合缩放+点击下钻 |
| 底部功能栏 | ✅ 已有模式 | 标准移动端导航模式（4个功能按钮+头像） |
| 骑行自动转化攻击力 | 🆕 创新模式 | 类似微信读书的"阅读→书币"，但更具象化 |
| 地图领土着色实时变化 | 🆕 创新模式 | 无直接参考，需要自行设计视觉语言 |
| 异步协作攻城 | 🔄 改造模式 | 从部落冲突借鉴，但去掉主动操作 |
| 周循环仪式感 | 🔄 改造模式 | 从赛季制游戏借鉴，缩短为周循环 |

**创新模式的用户教育策略（不使用教程）：**

| 创新模式 | 教育方式 | 原则 |
|---|---|---|
| 骑行→攻击力转化 | 第一次骑行后的贡献浮层直接展示因果关系 | "用结果说话，不用规则解释" |
| 领土着色变化 | 首屏地图上已有战队颜色，用户自然理解"颜色=占领" | "视觉自解释" |
| 周循环机制 | 顶部常驻状态条显示"对战期 · 剩余3天" | "环境暗示，不主动教学" |

### Experience Mechanics

**核心交互的逐步机制设计：**

**场景1：骑行后打开H5（最高频场景）**

```
步骤1 - 启动 (0-500ms)
├── 触发：用户在顽鹿App内点击"骑行征服"入口
├── 显示：深色骨架屏 + 地图轮廓渐显动画
└── 感受：「科技地理」氛围启动

步骤2 - 贡献浮层 (500ms-2s)
├── 触发：系统检测到本次有新骑行数据
├── 显示：底部浮层滑入 ──────────────────
│         │ 🚴 本次骑行 23.5 km              │
│         │ ⚔️ 攻城进度 52% → 58% (+6%)     │
│         │ 📍 目标：浦东新区                 │
│         │ [查看地图]                        │
│         ──────────────────────────────────
├── 交互：点击[查看地图]自动缩放到目标县区
└── 感受：「我的努力被看到了」

步骤3 - 地图浏览 (用户主动)
├── 触发：浮层消失后 或 用户点击[查看地图]
```

### Accessibility Considerations

**深色UI的特殊可访问性要求：**

**1. 对比度标准（WCAG 2.1 AA）**

| 元素 | 最低对比度 | 我们的实际对比度 | 状态 |
|---|---|---|---|
| 正文文字 vs 背景 | 4.5:1 | 5.8:1 (`#94A3B8` on `#0A1628`) | ✅ 达标 |
| 标题文字 vs 背景 | 3:1 | 15.4:1 (`#F1F5F9` on `#0A1628`) | ✅ 远超标准 |
| 按钮文字 vs 按钮色 | 4.5:1 | 8.6:1 (`#FFFFFF` on `#3B82F6`) | ✅ 达标 |
| 领土边界线 | 3:1 | 需动态检测（半透明色） | ⚠️ 需测试验证 |

**2. 触控可访问性**

- 所有可交互元素最小44px×44px（iOS HIG标准）
- 相邻可点击元素间距≥8px，防止误触
- 地图缩放：支持双指缩放和双击缩放两种方式
- 浮层：支持下滑关闭和点击遮罩关闭两种方式

**3. 动画可访问性**

- 尊重系统`prefers-reduced-motion`设置
- 减弱动画模式下：脉冲光效→静态高亮、过渡动画→即时切换
- 所有动画持续时间不超过1秒（骨架屏脉冲除外，但属于加载态）
- 无闪烁动画（避免光敏癫痫触发）

**4. 信息传达不依赖单一感官**

- 领土状态不仅用颜色区分，还有文字标签（"攻击中"/"已占领"/"防守中"）
- 进度不仅用颜色变化，还有百分比数字
- 倒计时不仅用视觉，还在关键节点提供提示文案

## Design Direction Decision

### Chosen Direction: 横屏沉浸地图布局

**核心定位**：横屏游戏化布局，营造「战略指挥屏」沉浸体验。地图占据 85% 屏幕，顶部信息条 + 底部功能栏固定布局。

**布局结构（1920x1080 基准）：**

```
┌────────────────────────────────────────────────────┐
│  顶部信息条 (80px)                                  │ 固定顶部
│  Logo「骑迹大陆」 + 周循环状态条                    │
├────────────────────────────────────────────────────┤
│                                                    │
│                                                    │
│          【中国地图 - 85% 屏幕高度】                 │ 地图主视觉
│          高度约 900px                               │ 可缩放、可点击
│          2800 个县区领土着色                          │ 手势缩放 + 点击下钻
│                                                    │
│                                                    │
├────────────────────────────────────────────────────┤
│  底部功能栏 (100px)                                 │ 固定底部
│  [战队] [作战] [战报] [排行]      👤 头像          │ 4 个功能按钮 + 头像
└────────────────────────────────────────────────────┘
```

### 选择横屏布局的核心理由

**1. 沉浸感最大化**
- 横屏 1920x1080 提供更大画布，地图占据 85% 屏幕
- 「战略指挥屏」视觉氛围需要宽度来展现全国领土的宏观态势
- 与 PRD 的「卫星夜景图 + 游戏化」定位完全一致

**2. 信息层次清晰**
- 顶部信息条：品牌 Logo + 周循环状态（固定，不占用地图空间）
- 中部地图区：完整中国地图，2800 个县区可清晰标注
- 底部功能栏：4 个核心功能 + 头像，符合用户拇指操作习惯

**3. 游戏化体验**
- 横屏是策略游戏（SLG）的标准布局（参考《率土之滨》《部落冲突》）
- 用户心智中「横屏=游戏」的认知已成定式
- 横屏布局天然营造「玩游戏」而非「用 App」的心理预期

**4. 技术实现友好**
- 固定布局（顶部 + 中部 + 底部）简化了响应式适配
- 1920x1080 基准尺寸，横屏锁定适配移动端WebView
- Vue 3 + ECharts 技术栈成熟支持横屏布局

### 横屏布局规范

| 维度 | 横屏布局（采用） |
|------|----------------|
| 地图可视面积 | 85% 屏幕 |
| 沉浸感 | 强（游戏化） |
| 单手操作 | 需双手（横屏模式） |
| 用户心智 | 游戏 |
| 响应式复杂度 | 低（横屏锁定+等比缩放） |

**横屏锁定要求：**
- WebView 强制横屏锁定（1920x1080 基准）
- 移动端竖屏状态禁止进入，提示"请横屏体验征服地图"

### 实施要点

**地图交互**：
- 支持手势缩放（双指捏合）、拖动、点击县区
- 三级下钻：全国视图 → 省份视图 → 县区详情浮层
- 家乡县区自动高亮（金色脉冲效果）

**顶部信息条**：
- 左侧：Logo「骑迹大陆」+ Slogan
- 右侧：周循环状态条（48px 高，圆角 12px）
- 固定定位，不随页面滚动

**底部功能栏**：
- 4 个功能按钮：战队、作战、战报、排行（各 100px 宽）
- 右侧：用户头像（带金色旋转光环）
- 固定定位，背景毛玻璃效果（`backdrop-filter: blur(20px)`）

**浮层系统**：
- 浮层宽度：800px（max），90%（响应式）
- 浮层高度：80vh（max）
- 圆角：16px
- 背景：`rgba(26, 39, 64, 0.95)` + 毛玻璃

## User Journey Flows

### 旅程A：新用户首次体验流

**目标**：30秒内完成情感连接，从「这是什么？」到「我的骑行跟家乡有关系了」
**来源**：PRD Journey 4（新用户30秒定生死）

**交互流程：**

1. **用户点击「骑行征服」入口** → 检测登录状态（复用顽鹿账户，无需额外注册）
2. **Layer 0 骨架屏**（<500ms）→ 深色背景 + 脉冲动画
3. **Layer 1 简化地图**（<1.5s）→ CDN静态全国地图加载
4. **地图自动飞到用户位置** → 动画缩放到家乡县区
5. **大字显示家乡状态**（三种文案之一）：
   - 「XX区正在被YY战队守护」
   - 「XX区正遭受ZZ战队攻击！」
   - 「XX区还是无主之地」
6. **引导第一次行动**：
   - 有战队用户：「你的战队正在攻打XX区，进度58%，你的历史骑行可贡献23km」→ [立即贡献]
   - 无战队用户：「你上次的骑行（18km）可以为家乡贡献守护力量」→ [贡献骑行]
7. **点击贡献** → 地图亮光反馈 + 进度条跳动 + 数字变化
8. **Layer 2 完整交互地图加载** → 底部功能栏出现 → 用户进入正常浏览

**关键设计决策：**

- **先行动后理解**：用户第一次交互是「贡献骑行」而非「阅读规则」
- **渐进加载掩盖延迟**：三层渲染让用户在等待时已经产生情感连接
- **家乡定位是锚点**：所有文案围绕「你的家乡」而非「游戏规则」
- **30秒承诺**：从点击入口到完成第一次贡献，不超过30秒

### 旅程B：骑行后反馈流

**目标**：骑行结束后打开游戏，立即看到贡献影响
**来源**：PRD Journey 1（战队骑友核心循环）

**交互流程：**

1. **用户完成骑行** → 顽鹿运动记录数据
2. **打开骑行征服** → 检测到新骑行数据未同步
3. **顶部横幅通知**：「你的40km已计入攻城！」（非模态，3秒自动消失，带短暂高亮动画）
4. **根据当前模式展示反馈**：
4. **横屏布局反馈**：
   - **地图区域**：攻击目标县区脉冲光效 + 进度数字跳动「宝安区 62% → 67%」
   - **顶部信息条**：短暂高亮显示"+67% 进度"徽章（1 秒后淡出）
   - **底部功能栏**：战队按钮角标更新「还差 330km」
5. **用户可继续浏览** → 缩放地图查看其他战区 / 点击底部功能按钮查看战队/作战/战报

**关键设计决策：**

- **入口即反馈**：打开游戏的第一个视觉信号就是「你的贡献已计入」
- **横屏优势**：地图占据 85% 屏幕，领土颜色变化更醒目
- **情感铁律「每骑必响」**：无论用户从哪个入口进入，都能在 3 秒内看到贡献结果
- **自然引导**：不弹模态框，用顶部横幅通知 + 地图高亮，不打断浏览节奏

### 旅程C：会长作战决策流

**目标**：会长在30秒内完成周作战计划确认
**来源**：PRD Journey 3（会长零管理成本）

**交互流程：**

1. **宣战窗口开始**（周日12:00）→ 推送通知：「本周战报已出炉，请确认新一轮作战计划」
2. **会长打开骑行征服** → 作战按钮角标显示「待确认」红点
3. **点击底部「作战」按钮** → 作战浮层从底部展开（宽度 800px，高度 80vh）
4. **顶部信息条显示**：「🎯 宣战决策 · 剩余 23h」
4. **系统智能推荐卡片**：
   ```
   📋 推荐作战计划
   攻击：锦江区（弱防守·距离近）
   维持：武侯区（已占领·城防82%）
   [✅ 一键确认]  [✏️ 自定义]
   ```
5. **三种路径**：
   - **一键确认**（默认路径，<15秒）→ 作战计划自动公示给战队成员
   - **自定义**（高级路径）→ 展开可攻击目标列表（按推荐度排序）→ 选择 → 预览 → 确认
   - **不操作**（兜底路径）→ 周一12:00系统推荐计划自动生效，零惩罚

**关键设计决策：**

- **浮层高效交互**：作战浮层从底部展开，不离开地图上下文，会长可以看到领土分布做决策
- **智能推荐 + 一键确认**：默认路径是「看一眼 → 确认」，不超过15秒
- **自定义是可选的**：高级操作不在默认路径上，避免增加认知负担
- **零操作兜底**：会长忘了操作也不会惩罚战队，系统自动执行推荐方案
- **情感铁律「紧迫不焦虑」**：用「剩余23h」而非闪烁倒计时

### 旅程D：周结算与战报查看流

**目标**：结算后用户看到本周战果，产生分享冲动
**来源**：PRD Journey 1, 3, 7（周循环结算）

**交互流程：**

1. **周日12:00 战报发布** → 推送通知：「本周战报：深圳闪电征服宝安区！🎉」
2. **用户打开骑行征服** → 自动导航到战报Tab
3. **战报全屏展示**：
   ```
   🏆 深圳闪电 · 第3周战报
   ✅ 攻下：宝安区（新领土！）
   🛡️ 守住：南山区（城防75%）
   📈 战队总里程：2,840km
   👤 你的贡献：110km（#2）
   ```
4. **领土变化动画**：地图上宝安区从灰色渐变为战队颜色 + 脉冲光效
5. **个人荣誉卡片**：「老周 · 本周贡献110km · 队内排名#2 · 攻城功臣」
6. **用户操作**：
   - 分享 → 自动生成分享卡片（战队名 + 领土数 + 个人贡献）→ 调用系统分享
   - 查看详情 → 展开每日贡献曲线、队友排名、攻防进度时间线
   - 返回地图 → 新领土已标记战队颜色

**关键设计决策：**

- **推送直达战报**：不需要用户自己找，打开就是结果
- **叙事顺序**：战队成果 → 领土动画 → 个人荣誉（先集体后个人）
- **分享素材自动生成**：降低分享门槛，用户不需要截图裁剪
- **情感铁律「集体叙事优先」**：战报标题是「深圳闪电征服宝安区」而非「你贡献了110km」

### 旅程E：异常与边缘场景处理流

**目标**：异常情况下用户不困惑、不焦虑
**来源**：PRD Journey 5（边缘案例）

**场景1：换战队**

1. 用户退出原战队 → 提示：「已贡献的里程归属不变，不会被撤回」
2. 加入新战队 → 提示：「从本周起，你的里程为新战队效力」
3. 心智模型：员工离职 —— 之前的工作成果留在原公司

**场景2：GPS数据异常**

1. 骑行数据同步 → 系统检测异常（速度>60km/h持续1h 或 单次里程超合理阈值）
2. 数据标记「核实中」，暂不计入攻防
3. 用户看到：「本次骑行数据核实中，预计24小时内确认」
4. 运营复核 → 正常则恢复计入并通知 / 异常则剔除并说明原因

**场景3：长期未骑行**

1. 用户一周未骑行 → 不发惩罚通知（铁律：永远正面表达）
2. 下次打开时显示：「战队在你不在时守住了XX区💪 欢迎回来继续战斗」

**异常处理设计原则：**

- **正面重述**：把负面事实用正面方式表达
- **过程透明**：显示「处理中」状态而非沉默
- **类比解释**：用生活类比解释规则（换战队=离职）
- **所有异常文案**：用陈述句而非警告句，用进行时而非过去时

### Flow Optimization Principles

**1. 最短路径到价值（Steps-to-Value）**

| 旅程 | 从打开到核心价值 | 最少步骤 |
|---|---|---|
| 新用户首次体验 | 看到家乡 + 完成第一次贡献 | 2步（自动定位 → 点击贡献） |
| 骑行后反馈 | 看到贡献影响 | 0步（打开即可见横幅通知） |
| 会长作战决策 | 确认作战计划 | 1步（一键确认推荐） |
| 战报查看 | 看到本周战果 | 0步（推送直达战报页） |

**2. 认知负荷控制**

- 每个界面最多3个操作选项：主操作（高亮）+ 次操作 + 返回/关闭
- 信息渐进展示：概览 → 点击展开详情 → 再点击看完整数据
- 智能默认值：系统替用户做80%的决策，用户只需确认

**3. 错误预防优于错误恢复**

- 会长宣战不设「撤回」，而是用确认预览防止误操作
- 不提供「退出战队」的快捷入口，放在设置深处减少冲动操作
- GPS异常自动检测，不等用户发现后投诉

**4. 情感连续性**

- 每次打开都有新信息（进度变化、队友动态），杜绝「打开后什么都没变」
- 关闭再打开恢复上次浏览位置和模式
- 推送通知文案和应用内文案保持叙事一致性

## UX Consistency Patterns

### Button Hierarchy

**三级按钮体系（每个界面最多展示3个层级）：**

| 层级 | 样式 | 使用场景 | 规范 |
|---|---|---|---|
| **主操作（Primary）** | 实心填充 `#3B82F6`，白色文字，圆角8px | 每个界面唯一的核心操作 | 每屏最多1个；高度48px；全宽或最小宽度120px |
| **次操作（Secondary）** | 描边 `#3B82F6`，蓝色文字，圆角8px | 可选的替代操作 | 每屏最多2个；高度40px |
| **文字操作（Tertiary）** | 无边框，蓝色文字，下划线 | 辅助链接、「查看更多」 | 不限数量；高度32px；最小点击区域44px |

**按钮状态规范：**

| 状态 | 视觉变化 | 触发条件 |
|---|---|---|
| Default | 标准样式 | 初始状态 |
| Pressed | 透明度降至0.7 + 轻微缩放0.98 | 手指按下 |
| Loading | 文字替换为旋转图标 + 禁用交互 | 操作处理中 |
| Disabled | 透明度0.4 + 禁用交互 | 条件不满足 |
| Success | 绿色闪烁0.3秒后恢复 | 操作成功 |

**特殊按钮：**
- **一键确认按钮**（会长宣战）：主操作按钮 + ✅图标前缀，强调「一步完成」
- **贡献按钮**（首次体验）：主操作按钮 + 脉冲边框动画，引导注意力

### Feedback Patterns

**四种反馈类型：**

| 类型 | 触发场景 | 视觉样式 | 持续时间 | 交互 |
|---|---|---|---|---|
| **成功** | 贡献计入、宣战确认、加入战队 | 绿色横幅 `#22C55E` + ✅ | 3秒自动消失 | 点击可跳转详情 |
| **信息** | 数据同步中、战报发布提醒 | 蓝色横幅 `#3B82F6` + ℹ️ | 3秒自动消失 | 点击可跳转 |
| **警告** | 宣战窗口即将关闭、城防偏低 | 琥珀色横幅 `#F59E0B` + ⚠️ | 需手动关闭 | 附带操作按钮 |
| **错误** | 网络断开、数据加载失败 | 红色横幅 `#EF4444` + 重试 | 需手动关闭或重试 | [重试]按钮 |

**反馈位置规范：**
- 顶部横幅：非阻断式通知（成功/信息类），从顶部滑入，不覆盖内容
- 页面内联：错误/空状态，在内容区域原位展示
- 底部浮层：需要用户决策的确认类反馈

**反馈优先级规则：**
- 同时多个反馈时，按 错误 > 警告 > 成功 > 信息 排队
- 同类型反馈合并（如连续2次骑行同步 → 合并为一条通知）
- 任何反馈不遮挡底部功能栏

| 战队 | 👥 | 战队 | TeamView |

- 底部功能栏固定4个功能按钮 + 头像
- Tab间切换无过渡动画（即时切换，保持响应感）
- 每个Tab记住自己的滚动位置和子状态
- 当前Tab再次点击 → 滚动回顶部
- 未读消息用红点(BaseBadge)标记在图标右上角

**二级导航：**

| 模式 | 实现 | 使用场景 |
|---|---|---|
| 模式切换 | 顶部胶囊Tab（地图/列表） | MapView内 |
| 浮层导航 | 从底部滑出半屏浮层，手势下滑关闭 | 县区详情、成员详情 |
| 返回导航 | 左上角返回箭头或手势右滑 | 从详情页返回 |

**页面跳转规范：**
- Tab切换 → 无动画
- 进入详情 → 从右侧滑入（300ms ease-out）
- 弹出浮层 → 从底部滑入 + 遮罩淡入（250ms）
- 浮层关闭 → 下滑手势或点击遮罩（200ms）

### Loading & Empty States

**加载状态分级：**

| 场景 | 加载方式 | 视觉表现 |
|---|---|---|
| 首次加载（冷启动） | 三层渐进渲染 | 骨架屏 → 简化地图 → 完整地图 |
| 页面切换 | 骨架屏 | BaseSkeleton脉冲动画，≤500ms |
| 数据刷新 | 内容保留 + 指示器 | 顶部进度条（细线蓝色），不替换内容 |
| 操作等待 | 按钮Loading状态 | 按钮内旋转图标，禁用交互 |
| 下拉刷新 | 顶部拉出指示器 | 旋转箭头 → 刷新动画 → 收回 |

**空状态规范：**

| 场景 | 文案 | 操作 |
|---|---|---|
| 无战队 | 「加入一支战队，开始你的征服之旅」 | [浏览战队] |
| 无攻防任务 | 「本周作战尚未开始，关注宣战公告」 | [查看时间表] |
| 无战报 | 「第一份战报将在本周日发布」 | — |
| 网络断开 | 「网络开小差了，试试重新连接」 | [重试] |
| 数据加载失败 | 「加载遇到问题，点击重试」 | [重试] |

空状态文案永远正面表达，引导下一步而非描述问题。图标用简笔线条风格。

### Modal & Overlay Patterns

**浮层类型：**

| 类型 | 高度 | 使用场景 | 关闭方式 |
|---|---|---|---|
| 半屏浮层 | 50%-70%屏幕 | 县区详情、成员详情、攻防详情 | 下滑手势 / 点击遮罩 |
| 底部操作栏 | ≤200px | 快捷操作、模式切换 | 点击遮罩 / 选择后自动关闭 |
| 全屏浮层 | 100%屏幕 | 战报详情、作战计划详情 | 关闭按钮 / 右滑手势 |
| 确认对话框 | 内容自适应 | 退出战队、发起攻击确认 | 确认/取消按钮 |

**浮层规范：**
- 遮罩 `rgba(0, 0, 0, 0.6)`，点击遮罩关闭（确认对话框除外）
- 浮层背景 `#1E293B`（深色主题第二层级）
- 半屏浮层顶部显示拖拽指示条
- 同时最多1个浮层，新浮层替换旧浮层

### Animation & Transition Patterns

**动画分类及时长：**

| 类型 | 时长 | 缓动函数 | 使用场景 |
|---|---|---|---|
| 即时反馈 | 100-150ms | ease-out | 按钮按下、Tab切换 |
| 界面过渡 | 200-300ms | ease-out | 页面滑入、浮层弹出 |
| 数据动画 | 400-600ms | ease-in-out | 进度条变化、数字跳动 |
| 情感动画 | 600-1000ms | ease-in-out | 领土颜色变化、脉冲光效 |
| 引导动画 | 持续 | linear | 骨架屏脉冲、按钮呼吸 |

**动画铁律：**
1. 尊重 `prefers-reduced-motion` —— 减弱动画模式下所有非功能性动画禁用
2. 所有动画可被用户操作打断
3. 数据动画优先于装饰动画（网络慢时先更新数字，再播放光效）
4. 动画不阻塞交互 —— 动画播放中用户仍可点击操作

## Responsive Design & Accessibility

### Responsive Strategy

**平台定位：纯移动端 H5（WebView 内嵌）**

本产品运行在顽鹿运动App的 WebView 中，不存在独立浏览器访问场景，因此：

- ❌ 不需要桌面端适配
- ❌ 不需要平板端适配
- ✅ 仅需覆盖主流手机屏幕（320px - 428px 宽度范围）

**设计基准：375px（iPhone 逻辑分辨率）**

| 适配策略 | 规则 |
|---|---|
| 文字/按钮/间距 | 固定 px 值（不随屏幕缩放） |
| 容器/卡片/列表 | Flex 弹性布局，宽度 `fill_container` |
| 地图区域 | 宽度 100vw，高度 70vh（地图模式）或 0（列表模式） |
| 底部导航栏 | 固定定位，宽度 100%，适配 `safe-area-inset-bottom` |

**核心原则：**
1. 内容优先于装饰——小屏优先保证信息可读性
2. 触控优先——所有可交互元素 ≥ 44px 触控热区
3. 弹性容器 + 固定元素——框架弹性伸缩，文字和按钮保持固定尺寸

### Breakpoint Strategy

**单断点策略（极简）**

由于目标设备范围极窄（320-428px），仅设一个断点：

| 断点 | 范围 | 策略 |
|---|---|---|
| 小屏适配 | ≤ 374px | 压缩非关键间距，BattleCard 切换为紧凑模式 |
| 标准布局 | ≥ 375px | 设计基准，所有组件按标准尺寸渲染 |

**小屏适配细节（≤ 374px）：**
- 卡片内间距从 16px 降为 12px
- 地图模式操作栏高度从 30vh 降为 25vh
- 标签文字从 14px 降为 12px
- 底部 Tab 图标从 24px 降为 20px

**不需要的：**
- ❌ 多列布局断点（始终单列）
- ❌ 大屏弹性放大（最大宽度 428px 已覆盖所有 iPhone/Android 主流机型）

**横屏锁定：**
- ✅ WebView 横屏锁定，确保 1920x1080 基准布局
- ✅ 移动端竖屏状态禁止进入，提示"请横屏体验征服地图"

### Accessibility Strategy

**目标合规等级：WCAG 2.1 AA**

本产品虽然是游戏化场景，但作为面向大众骑行用户的产品，需要保证基本的无障碍访问能力。

**色彩无障碍（三重编码原则）：**

所有依赖颜色传递信息的场景，必须同时提供三重编码：

| 编码层 | 说明 | 示例 |
|---|---|---|
| 颜色 | 视觉主通道 | 我方领土蓝色、敌方红色 |
| 文字标签 | 明确语义 | 「我方」「敌方」「争夺中」 |
| 图标/形状 | 辅助区分 | 盾牌=防守、剑=攻击、旗帜=已占领 |

**色盲支持：**
- 蓝/橙对比代替红/绿对比（色盲友好色对）
- 领土着色叠加纹理图案（斜线=我方，点阵=敌方）—— 作为可选的「无障碍模式」
- 进度条增加百分比数字标注

**触控无障碍：**

| 规则 | 标准 |
|---|---|
| 最小触控目标 | 44 × 44px |
| 触控间距 | 相邻可点击元素间距 ≥ 8px |
| 手势替代 | 所有滑动手势均提供按钮替代方案 |
| 双击保护 | 关键操作（宣战、退出）按钮 300ms 防抖 |

**动效无障碍：**
- 尊重 `prefers-reduced-motion` 系统设置
- 减弱动画模式下：禁用脉冲光效、领土渐变动画，保留功能性过渡（页面滑入、数字更新）
- 地图模式提供列表模式作为完全等价的无障碍替代方案

**屏幕阅读器支持：**
- 所有动态内容区域添加 `aria-live` 属性
- 地图交互提供文字描述（"当前查看：北京市朝阳区，我方占领，贡献度 85%"）
- 倒计时组件添加 `aria-label`（"距离本周结算还剩 2天 14小时"）
- 模式切换 Tab 使用 `role="tablist"` + `aria-selected`

### WebView Specific Adaptations

**安全区域适配：**
```
padding-bottom: env(safe-area-inset-bottom);
padding-top: env(safe-area-inset-top);
```

**WebView 交互优化：**

| 优化项 | CSS/JS 处理 |
|---|---|
| 禁止页面弹性滚动 | `overscroll-behavior: contain` |
| 禁止双击缩放 | `touch-action: manipulation` |
| 禁止文字选中 | `user-select: none`（输入框除外） |
| 禁止长按菜单 | `-webkit-touch-callout: none` |
| 字体大小稳定 | `-webkit-text-size-adjust: 100%` |

**横屏适配（WebView 横屏锁定）：**
- WebView 横屏锁定，确保 1920x1080 基准布局
- 移动端竖屏状态禁止进入，提示"请横屏体验征服地图"

**状态栏融合：**
- 深色主题下状态栏使用 `#0A1628`（与背景融合）
- WebView 顶部不预留额外空间，由 App 原生层控制

- 动画帧率 ≥ 30fps（减弱动画模式下无需测试动画性能）

### Implementation Guidelines

**开发清单（Developer Checklist）：**

1. ✅ 所有页面使用 `<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no">`
2. ✅ 底部导航栏和浮层底部添加 `safe-area-inset-bottom` 适配
3. ✅ 所有可交互元素最小尺寸 44 × 44px
4. ✅ 文字颜色与背景色对比度 ≥ 4.5:1（大文字 ≥ 3:1）
5. ✅ 地图组件的领土信息提供等价文字描述
6. ✅ 所有表单元素关联 `<label>` 或 `aria-label`
7. ✅ 使用语义化 HTML：`<nav>`, `<main>`, `<section>`, `<button>`
8. ✅ 动态内容更新使用 `aria-live="polite"`（紧急通知用 `assertive`）
9. ✅ CSS 动画包裹在 `@media (prefers-reduced-motion: no-preference)` 中
10. ✅ 小屏断点（≤374px）下验证所有组件无溢出截断
