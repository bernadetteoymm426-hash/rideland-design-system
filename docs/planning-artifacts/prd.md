# Product Requirements Document - Project 001

**Author:** Mark
**Date:** 2026-03-07

## Executive Summary

顽鹿骑行征服（暂定名）是一款嵌入顽鹿运动App的H5骑行征服游戏，将用户真实骑行里程转化为全国领土争夺的驱动力。玩家以骑行战队为单位，在全国约2800个县区级行政区域地图上攻城掠地——每一公里骑行都直接推动领土攻防进度。

本产品解决的深层问题是：**个体运动天然孤独、缺乏持续的群体目标感**。传统骑行社交依赖线下组队，组织成本高、频次低。本产品通过异步协作（无需同时骑行）和持续目标（每周攻防永不停歇）的机制，让每一次独自骑行都服务于集体征服目标，不增加组织成本却显著增强社交粘性。

目标用户群体覆盖顽鹿运动平台15万MAU。其中约8.3%（~12,500人）已有战队归属，为Alpha阶段的核心竞争玩家；91.7%的散客用户将在Beta阶段通过弱贡献机制（为家乡区域贡献里程）参与游戏，并作为战队拉新的转化漏斗。产品预期拉动用户骑行量增长、提升App日活与留存、增强战队组织活力，并为超级个体平台模式提供首个标杆级验证案例。

产品以三阶段上线：Alpha（50个核心战队内测，验证核心闭环）→ Beta（200个战队公测，完善散客和运营体系）→ GA（全量开放）。

### What Makes This Special

**骑行即征服——每一公里都在改变全国领土版图。**

区别于"跑步换积分""骑行换金币"的虚假货币转换模式，本产品将骑行里程**直接映射为领土攻防力量**——用户不是在兑换抽象积分，而是在真实推动地图上的攻城进度条。这种映射更直接、更可视化、更能激发持续行为。

核心差异化体验：
- **家乡归属感**：用户首次打开即定位到所在区域，看到"你的家乡正在被XX战队守护"或"你的家乡还是无主之地"——情感冲击先于规则理解
- **每骑必反馈**：每次骑行结束即可看到"你的23km推动了攻城进度从52%到58%"——骑行第一次有了超越健康之外的意义
- **集体叙事**：从碎片县区到统一地级市、统一省份的逐级合并展示，战队的扩张历程自然生成史诗叙事——"XX铁骑统一山东省"本身就是社交传播素材
- **零管理成本协作**：会长操作<30秒（智能推荐+一键确认+不操作自动兜底），让异步团队协作真正可持续

## Success Criteria

### 北极星指标

**核心度量：参与游戏的用户周均骑行天数提升量。** 这是骑行征服存在的根本理由——如果用户不因为游戏多骑车，产品就失败了。所有其他指标服务于这一核心。

### User Success

| 成功标准 | 衡量方式 | 目标值 |
|---|---|---|
| **骑行频次提升** ⭐ | 参与游戏的用户周均骑行天数 vs 参与前 | 周均骑行天数 +1天以上 |
| **非骑行时间回访** | 用户在非骑行时段（如通勤、午休）打开游戏页面的比例 | 日均回访用户占周活跃用户的 ≥40% |
| **战队周活跃率** | 每周至少贡献1次骑行里程的成员占比 | ≥50% |
| **首次体验转化** | 新用户首次打开到完成第一次有效互动（查看家乡/加入战队/贡献里程）的比例 | ≥60%（30秒内产生情感连接） |
| **会长操作零负担** | 会长完成每周作战计划设置的平均耗时 | ≤30秒（含智能推荐一键确认） |
| **用户满意度** | 游戏内NPS/满意度评分（1-10） | ≥7 |
| **竞争平衡感知** | Alpha结束时用户调研："竞争是否公平"正面回答比例 | ≥60% |

### Business Success

| 成功标准 | 衡量方式 | 目标值 |
|---|---|---|
| **AI开发模式验证** | 非专业开发团队（Mark+AI）能否完成从PRD到上线的完整流程 | 成功上线并稳定运行 |
| **产品完整度** | 核心功能全部可用、无阻塞性bug、用户体验流畅 | Alpha阶段：核心攻防循环完整可玩 |
| **用户使用稳定性** | 上线后连续4周无重大故障（P0/P1级bug） | 99.5%可用性 |
| **用户满意度验证** | Alpha用户满意度评分 | ≥7/10 |
| **超级个体平台标杆** | 验证"个人+AI"模式可以产出上架质量的产品 | 通过内部审核并成功上架 |

> **明确边界**：活跃率增长、用户拉新、商业变现等指标属于**下一阶段**目标，本阶段不作为成功判定标准，但架构上预留数据采集能力。

### Technical Success

| 成功标准 | 衡量方式 | 目标值 |
|---|---|---|
| **系统稳定性** | 服务可用性（uptime） | ≥99.5% |
| **页面加载速度** | 首屏加载时间（含地图渲染） | P90 < 1秒 |
| **地图交互流畅度** | 2800个县区地图缩放/下钻操作帧率 | ≥30fps，操作响应 < 200ms |
| **API响应速度** | 核心接口（攻防计算、里程提交、战况查询）响应时间 | P95 < 500ms |
| **数据一致性** | 里程数据、领土归属、攻防进度的准确性 | 零数据丢失，最终一致性延迟 < 5分钟 |
| **并发承载** | Alpha阶段峰值并发用户数 | 支撑500+同时在线（50个战队规模） |

### Measurable Outcomes

Alpha阶段（50个战队内测）结束时的核心验证清单：

1. ✅ **可做性验证**：AI辅助开发模式成功完成"需求→设计→开发→上线"全流程
2. ✅ **可用性验证**：核心攻防循环完整可玩，用户满意度 ≥7/10
3. ✅ **稳定性验证**：连续4周运行，可用性 ≥99.5%，无P0故障
4. ✅ **体验性验证**：首屏加载 <1秒，地图交互流畅，会长操作 <30秒
5. ✅ **参与度信号**：战队周活跃率 ≥50%，用户产生非骑行时段回访行为

## Product Scope

### MVP — 最小可行产品（Alpha阶段）

核心原则：**完成一个完整的攻防循环，证明"骑行→领土变化"的核心体验成立。**

必须包含（9项）：

| # | 功能 | 说明 |
|---|---|---|
| 1 | **全国县区级地图** | ~2800个区域，ECharts三级下钻（全国→省→市/县区） |
| 2 | **战队体系** | 创建（1人创建/3人激活）、加入、退出、解散 |
| 3 | **大本营选择** | 首次设置战队所在区域，初始领土+情感锚点 |
| 4 | **骑行里程→攻防转化** | 基础里程直接转化为攻城/守城进度 |
| 5 | **宣战与攻防** | 公开宣战制 + 周循环（对战→结算→战报→宣战） |
| 6 | **领土可视化** | 半透明着色显示归属、攻城进度条 |
| 7 | **每骑即时反馈** | 骑行结束后展示里程贡献对攻防进度的影响 |
| 8 | **周战报** | 每周结算后的攻防结果推送 |
| 9 | **会长智能推荐作战计划** | 智能推荐 + 一键确认 + 不操作自动兜底（<30秒） |

> **首次体验引导**作为用户旅程中的交互设计需求（Journey 4），由UX阶段定义具体流程，不单独列为功能项。

明确排除（延后到Beta/Growth/Vision）：
- ❌ 散客弱贡献 — Alpha为受邀战队内测，散客场景有限（→Beta）
- ❌ 新手保护期 — Alpha受邀战队都是成熟队伍（→Beta）
- ❌ 半程战报 — 锦上添花，周战报已满足阶段性反馈（→Beta）
- ❌ 即时战役（室内限定晚间对战）
- ❌ 远征成就奖励
- ❌ 经销商特殊标识和联盟机制
- ❌ 骑行加成系数（功率/速度/爬升）
- ❌ 逐级统一成就系统（县区→地级市→省份合并）
- ❌ 周中随机事件
- ❌ 社交分享H5落地页
- ❌ 商业变现功能

### Growth Features（Beta阶段 — Post-MVP）

核心攻防验证通过后，丰富体验层和运营层：

1. **骑行加成系数** — 功率/速度/爬升的可选加成
2. **逐级统一成就** — 县区→地级市→省份合并展示 + 成就解锁
3. **周中事件填充** — 半程战报、每日挑战、随机事件
4. **叙事化战报** — 史诗风格的攻防叙事推送
5. **即时战役** — 室内骑行限定、晚间时段、1.5倍效果
6. **社交分享** — 独立H5分享落地页，支持裂变传播
7. **散客→战队转化漏斗** — 智能推荐加入、试用期体验
8. **接近胜利提醒** — 推送通知"再骑X公里即可拿下XX县"
9. **战队内贡献排行榜**
10. **数值平衡迭代** — 基于Alpha数据校准城防值、维护费

### Vision（GA及未来）

全量开放后的远景功能：

1. **经销商品牌领土** — 经销商战队特殊标识、品牌曝光、联盟机制
2. **远征成就** — 跨区域远征的特殊奖励体系
3. **赛季制** — 定期重置 + 历史战绩殿堂
4. **跨平台联动** — 与顽鹿竞技室内骑行数据打通
5. **商业变现** — 皮肤/特效/旗帜等虚拟物品、经销商广告位
6. **UGC地图事件** — 用户自定义区域事件/挑战
7. **AI对战策略** — AI辅助会长制定最优攻防方案

## User Journeys

### Journey 1: 战队骑友（核心玩家）— 成功路径

**人物**：老周，35岁，深圳某骑行战队资深成员，每周骑行3-4次，单次30-60km，偏好周末长距离拉练。

**Opening Scene — "又是一个人刷圈"**
周三晚上，老周在深圳湾独自骑了40公里。Strava记录、顽鹿运动同步——数据躺在那里，没人在意。战队群里偶尔有人发骑行截图，但大多数时候是沉默的。老周想：骑车这件事，到底跟别人有什么关系？

**Rising Action — "我的40公里居然有用"**
某天打开顽鹿运动，首页推送"骑行征服已上线"。点进去，地图自动定位到深圳南山区——"南山区目前无人占领"。老周的战队"深圳闪电"已经在游戏里了，会长把大本营设在南山。老周看到：深圳闪电正在攻打宝安区，进度条62%，离拿下还差380km。

老周今天骑的40km已经自动计入——进度条跳到67%。他翻了一下战队页面，发现队友小李今天也贡献了28km。两个人加起来推了5个百分点。

**Climax — "再骑一次就拿下了"**
周五晚上，老周本来打算休息。但打开游戏一看：宝安区攻城进度89%，还差约110km。群里会长发了一条："兄弟们，周末拿下宝安，深圳双区在手！"老周想了想，换上骑行服出门了——"就当恢复骑，骑个30公里。"

**Resolution — "骑车终于有了超越健康的意义"**
周六结算前，深圳闪电成功拿下宝安区。周日中午战报推送："深圳闪电本周征服宝安区，领土扩张至2块！老周本周贡献110km，排名队内第2。"老周截图发朋友圈，配文"打下宝安了"。三个骑友私聊问他这是什么游戏。

> **旅程揭示的能力需求**：骑行数据自动同步、攻城进度实时可视化、战队成员贡献展示、周结算战报推送、分享素材生成

### Journey 2: 散客骑友 — "我没有战队，但我也想玩"

> ⏸️ **阶段说明**：散客弱贡献机制延后至Beta阶段实现。本旅程描述Beta阶段的目标体验，Alpha阶段暂不支持散客独立参与。

**人物**：小陈，28岁，杭州独立骑行爱好者，每周骑2次通勤+周末偶尔长骑。没加过任何战队，觉得"组织太麻烦"。

**Opening Scene — "这游戏跟我有关系吗？"**
小陈在顽鹿运动看到"骑行征服"入口，点进去。地图自动定位到杭州西湖区。屏幕上写着："你的家乡西湖区目前被'杭州龙骑士'占领中。" 小陈有点意外——原来自己住的地方已经有人在争了。

**Rising Action — "我也能贡献？"**
页面下方提示："即使没有战队，你的骑行里程也可以守护家乡。"小陈今天通勤骑了12km，系统显示"你为西湖区贡献了12km守护力量"。虽然力量很小，但地图上西湖区亮了一下——有种微妙的参与感。

接下来几天，小陈每次骑完都会点进来看看：西湖区还在不在？有没有被别的战队攻打？他开始留意"杭州龙骑士"这个战队——他们在打滨江区了。

**Climax — "要不……我也加入试试？"**
两周后，系统推送："杭州龙骑士正在招募新成员，他们的目标是统一杭州全市。加入试试？"小陈犹豫了一下，点了"查看战队"——发现成员25人，平均年龄跟自己差不多，本周活跃18人。他点了"申请加入"。

**Resolution — "从路人变成战友"**
加入后第一次打开游戏，地图上多了队友的贡献轨迹，攻防计划里有了自己的名字。小陈的12km通勤现在不只是"守护家乡"，而是"攻打滨江区的一部分"。他第一次觉得每天的通勤骑行有了归属。

> **旅程揭示的能力需求**：散客弱贡献机制、家乡区域自动定位、散客→战队转化引导、战队信息展示与申请入口、个人贡献可视化

### Journey 3: 战队会长 — "我只想带队骑车，别让我当项目经理"

**人物**：张哥，42岁，成都"川西铁骑"会长，战队30人，核心活跃十几个。平时在群里组织骑行，但越来越觉得管理是个苦差事。

**Opening Scene — "又来一个要我操心的东西"**
收到顽鹿运动推送"骑行征服上线"。张哥第一反应是：又要花时间搞？上次搞战队活动报名表就折腾了半天。点进去看看再说。

**Rising Action — "居然不用我操心？"**
进入游戏，系统提示："您是川西铁骑的会长。系统已为您推荐大本营位置：成都武侯区（您和大多数成员的常驻地）。确认？"张哥点了确认。

接下来系统问："本周作战计划——系统推荐攻打锦江区（距离最近、防御最低）。确认？"张哥看了一眼，挺合理的，点了确认。整个过程不到15秒。

**Climax — "群里居然自己热闹起来了"**
周三，游戏自动在战队页面发布了"半程战报"：锦江区攻城进度45%，本周贡献排名前三是谁谁谁。张哥还没来得及在群里说什么，成员已经自己讨论开了："这周末我多骑一趟""锦江区稳了吧？"

张哥发现自己什么都不用做——系统自动推荐计划、自动发进度、成员自己就卷起来了。他要做的唯一一件事就是每周日下午在宣战窗口点一下"确认作战计划"。

**Resolution — "这个游戏帮我管战队"**
一个月后，川西铁骑的周活跃率从40%涨到65%。以前群里一周说不了几句话，现在每天都有人讨论"下一个打哪里"。张哥说："这不是游戏，这是战队管理工具。"

> **旅程揭示的能力需求**：会长智能推荐（大本营、作战计划）、一键确认操作、不操作自动兜底、半程战报自动推送、战队活跃度可视化

### Journey 4: 新用户（首次接触游戏）— "30秒定生死"

**人物**：小王，24岁，刚注册顽鹿运动不久，骑公路车半年，还没有加入任何战队，对平台功能不太熟悉。

**Opening Scene — "这是什么？"**
小王在顽鹿运动首页看到"骑行征服"的入口banner，出于好奇点了进去。

**Rising Action — 前30秒的体验**
- **第1秒**：全国地图加载，深色系背景上各色领土在发光
- **第3秒**：地图自动飞到小王的位置——湖南长沙岳麓区
- **第5秒**：大字显示"你的家乡：岳麓区"，下方状态："无人占领——这片土地在等待它的守护者"
- **第10秒**：提示"你上次的骑行（18km）可以为岳麓区贡献守护力量。立即贡献？"
- **第15秒**：小王点了"贡献"，地图上岳麓区微微亮了一下，显示"岳麓区 +18km 守护中"
- **第20秒**：页面展示"长沙地区有3个战队正在争夺领土"，列出战队名称和领土数
- **第30秒**：小王已经理解了核心概念——骑车→占地盘，而且他刚才已经"做了一件事"

**Climax — 从理解到参与**
小王发现自己作为散客也能为家乡贡献里程，开始每天骑完查看岳麓区的状态。三天后他点击了一个附近的战队"湘骑天下"，看到他们正在攻打雨花区。

**Resolution — 30秒内完成情感连接**
小王最后的感受不是"我学会了一个复杂游戏"，而是"我的骑行跟家乡有关系了"。规则可以慢慢学，但这种连接感在30秒内就建立了。

> **旅程揭示的能力需求**：极速加载首屏、地理位置自动定位、历史骑行数据回溯贡献、渐进式引导（先行动后理解）、零注册门槛（复用顽鹿账户）

### Journey 5: 战队骑友 — 边缘案例（异常与恢复）

**人物**：老李，38岁，北京某战队成员，遇到了各种"不顺利"的情况。

**场景A — "我换战队了，之前的贡献怎么办？"**
老李从"北京风行者"退出，加入了"北京闪电侠"。他之前为风行者贡献的200km还在那边的领土上。系统处理：已贡献的里程归属不变（已结算的历史不可撤回），但老李从本周起的新里程归属新战队。老李能理解——就像员工离职，之前的工作成果还是留在原公司。

**场景B — "我这周一公里没骑，战队会怎样？"**
老李出差一周，没有任何骑行。他的战队正在守城，城防每周衰减20%。老李回来发现丢了一块地。系统没有惩罚老李个人，只是客观呈现"本周未贡献里程"。他看到队友的贡献仍在维持其他领土——团队机制削弱了个人缺席的影响。

**场景C — "GPS数据异常，里程多算了"**
老李的一次骑行因为GPS漂移记录了300km（实际骑了30km）。系统有异常数据检测：单次骑行超过合理阈值（如平均速度>60km/h持续1小时）标记为异常，暂不计入攻防，后台脚本复核。

> **旅程揭示的能力需求**：成员变动时的里程归属规则、城防衰减机制、个人贡献缺失的容错设计、异常数据检测与过滤、数据复核流程

### Journey 6: 运营管理员（张雷团队）— "Alpha阶段脚本打天下"

**人物**：小刘，运营团队成员，负责骑行征服Alpha阶段的日常运营监控。

**Opening Scene — "数据在哪看？"**
Alpha上线第一天，小刘需要监控游戏运行状态。没有后台界面，他通过SQL查询和预置脚本来操作。

**Rising Action — 日常监控**
- **每日检查**：运行脚本查看当日活跃战队数、总骑行里程、异常数据条目
- **异常处理**：发现某用户里程数据异常（GPS漂移），通过脚本标记并剔除
- **周结算确认**：周六午夜结算自动执行后，运营在周日上午复核结果正确性——领土归属变更是否与攻防进度一致。如有问题，在周日中午正式发布前修正
- **用户反馈**：收到用户反馈"我的里程没算进去"，通过脚本查询该用户骑行记录和同步状态

**Climax — "有人疑似刷数据"**
发现某战队一夜之间里程暴增，小刘通过脚本查询该战队所有成员的骑行详情，发现一个账号在凌晨3点上传了多条异常骑行记录。小刘标记该账号数据为"待核实"，暂停其里程计入，通知张雷确认处理方案。

**Resolution — "Alpha够用了"**
小刘维护了一份运营脚本清单：日常监控、异常检测、数据修正、结算验证、用户查询。虽然不如后台界面方便，但50个战队的规模完全够用。他同时记录了"Beta阶段运营后台需求清单"，为下一阶段做准备。

> **旅程揭示的能力需求**：预置运营脚本集（监控、查询、修正）、异常数据标记接口、结算验证脚本、数据导出能力、运营操作日志

### Journey 7: 后端系统/自动化 — "无人值守的周循环"

**人物**：系统自动化流程（非人类角色，但需映射其交互路径）。

**周循环时序（正式时间表）**

| 时间段 | 阶段 | 系统行为 |
|---|---|---|
| **周一 12:00 — 周六 24:00** | ⚔️ 对战期（5.5天） | 接收骑行数据、计算攻防进度、实时更新领土状态 |
| **周六 24:00 — 周日 12:00** | 🔧 结算期（12小时） | 自动结算 + 运营复核窗口 |
| **周日 12:00** | 📢 战报发布 | 正式发布本周对战结果、推送战报 |
| **周日 12:00 — 周一 12:00** | 🎯 宣战窗口（24小时） | 会长决定新一轮进攻目标（不操作则自动兜底） |
| **周一 12:00** | 📣 宣战公布 | 公布所有战队宣战声明，新一轮对战期开始 |

**对战期（周一 12:00 — 周六 24:00）详细行为：**
- 持续接收顽鹿运动同步的骑行数据（通过内部API）
- 异常数据检测（速度/里程阈值校验）
- 实时计算里程→攻防进度转化
- 更新领土状态缓存（Redis）
- 触发"接近胜利"检测（进度>80%的攻城提醒）

**结算期（周六 24:00 — 周日 12:00）详细行为：**
- 周六 24:00：锁定本周所有骑行数据（截止计入）
- 自动计算所有攻城战结果（进度≥100%则攻城成功）
- 计算维护费（已占领区域线性递增）
- 更新领土归属
- 生成战报数据
- 运营团队复核窗口：如有异常数据或结算错误，运营可在周日12:00前修正

**宣战窗口（周日 12:00 — 周一 12:00）详细行为：**
- 周日 12:00：正式发布战报推送（通过顽鹿App推送通道）
- 会长进入宣战决策界面，系统提供智能推荐目标
- 会长确认或自定义宣战目标
- 周一 12:00前未操作的战队，系统自动执行兜底方案（推荐目标自动生效）
- 周一 12:00：公布所有宣战声明，应用城防衰减（所有已占领区域城防值 -20%），新对战期开始

**异常处理**
- 结算失败 → 自动重试3次 → 失败则告警运营（在12小时结算窗口内有足够时间处理）
- 数据同步延迟 → 队列积压监控 → 超阈值告警
- 并发冲突（同一区域多个战队同时攻打）→ 按进度优先、同进度按先宣战者优先

> **旅程揭示的能力需求**：定时任务调度系统、数据同步管道、结算引擎、推送通道对接、异常告警机制、并发冲突解决规则、Redis缓存更新策略、12小时运营复核缓冲机制

## Web App + Backend Service 项目类型需求

### Project-Type Overview

本产品为 **H5 Web App（前端）+ Golang Backend Service（后端）** 的混合架构，H5以SPA形式嵌入顽鹿运动App的WebView中运行，后端提供RESTful API服务。

核心特征：
- **非独立应用**：寄生于顽鹿运动App生态，复用其用户体系、骑行数据、推送通道
- **H5↔App通信**：通过JSBridge实现4个核心接口（用户认证、骑行数据获取、推送注册、分享调用）
- **H5↔后端通信**：标准RESTful API，JSON格式，HTTP/HTTPS

### Technical Architecture Considerations

#### 前端（H5 Web App）

| 维度 | 决策 | 说明 |
|---|---|---|
| **框架** | Vue 3 + Vite + TypeScript | 顽鹿团队现有技术栈 |
| **渲染模式** | SPA（单页应用） | WebView内运行，无需SSR/MPA |
| **地图引擎** | ECharts Map | 三级下钻（全国→省→市/县区），自带中国行政区划边界数据 |
| **浏览器兼容** | iOS 13+ WebView, Android 8+ WebView | 对应 Safari 13+, Chrome 70+ 内核 |
| **SEO** | 不需要 | App内嵌H5，非公开网页 |
| **无障碍** | 基础级别 | 确保文字可读、按钮可点击，不需要完整WCAG合规 |
| **离线支持** | 不需要 | 游戏依赖实时数据，离线无意义 |

#### 响应式设计

| 维度 | 决策 |
|---|---|
| **目标设备** | 手机竖屏为主（顽鹿运动App内使用场景） |
| **屏幕适配** | 移动端优先，适配 320px-428px 宽度范围 |
| **地图交互** | 支持手势缩放、拖拽、点击下钻 |
| **横屏** | 不主动适配，但不阻止（地图浏览时横屏体验更好可作为Growth优化项） |

#### 后端（Golang Backend Service）

| 维度 | 决策 | 说明 |
|---|---|---|
| **语言** | Golang | 顽鹿团队现有技术栈 |
| **数据库** | MySQL/PostgreSQL + Redis | 持久化存储 + 缓存/实时状态 |
| **API风格** | RESTful | 无需WebSocket，轮询满足实时性需求 |
| **数据格式** | 统一JSON | ECharts自带地图边界数据，后端只传业务数据 |
| **认证** | 复用顽鹿用户体系 | JSBridge传递token → 后端验证 |
| **API版本管理** | Alpha不加 | 前后端同步更新，路径直接用 `/api/xxx`，Beta阶段按需引入 `/api/v1/xxx` |
| **限流** | 简单全局限流 | 每用户每秒≤10次请求，Golang中间件实现 |

#### 错误处理（Error Handling）

| 维度 | 要求 | 说明 |
|---|---|---|
| **统一错误响应格式** | 所有API返回统一JSON错误结构 | `{ "code": <业务错误码>, "message": <用户可读描述>, "detail": <调试信息，仅非生产环境> }` |
| **业务错误码分类** | 按模块分段编码 | 1xxx=通用错误，2xxx=用户/认证，3xxx=战队，4xxx=攻防/骑行，5xxx=结算，6xxx=地图 |

| **关键操作重试** | 骑行贡献提交、宣战操作支持客户端自动重试 | 最多重试2次，间隔1秒，超时后提示用户手动重试 |
| **错误日志** | 所有5xx错误及业务异常写入结构化日志 | 包含 request_id、user_id、接口路径、错误码、堆栈摘要 |

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

**MVP类型：体验验证型（Experience MVP）**

本项目MVP的核心目标不是快速获客或商业变现，而是验证两件事：
1. **玩法验证**：骑行里程→领土攻防的核心循环是否能持续激发骑行动力
2. **模式验证**：一个人+AI能否独立完成从产品设计到开发上线的完整流程，为超级个体平台树立标杆

**资源配置：** Mark（产品+全栈开发）+ AI辅助开发，不依赖顽鹿研发团队
**上线策略：** Alpha内测 → 50个核心战队受邀参与 → 验证核心闭环后扩展

### MVP Feature Set (Phase 1 - Alpha)

**核心用户旅程支持：**
- ✅ 战队会长作战旅程（宣战→智能推荐→确认→追踪→结算）
- ✅ 战队成员骑行贡献旅程（骑行→即时反馈→看到领土变化）
- ✅ 新用户首次体验旅程（打开→看到家乡→情感连接→加入战队）
- ⏸️ 散客贡献旅程 → Beta阶段

**Must-Have 9项功能：**

| # | 功能 | 必要性 | 验证目标 |
|---|---|---|---|
| 1 | **全国县区级地图** | 游戏载体，没有地图没有游戏 | 2800区域渲染+三级下钻可用性 |
| 2 | **战队体系** | 核心组织单位 | 创建（1人建/3人激活）、加入流程 |
| 3 | **大本营选择** | 初始领土+情感锚点 | 首次体验即产生归属感 |
| 4 | **骑行里程→攻防转化** | 核心闭环的动力引擎 | 每公里骑行都有意义 |
| 5 | **宣战与攻防** | 核心闭环的策略层 | 公开宣战+周循环运转正常 |
| 6 | **领土可视化** | 核心闭环的反馈层 | 领土变化清晰可感知 |
| 7 | **每骑即时反馈** | 骑行动力的关键闭环 | 骑完立即看到攻城进度变化 |
| 8 | **周战报** | 阶段性成就感+叙事感 | 每周有完整的"战争故事" |
| 9 | **会长智能推荐作战计划** | 零管理成本的关键保障 | 推荐+一键确认+自动兜底<30秒 |

**Alpha明确排除（延后到Beta/Growth）：**

| 功能 | 延后原因 |
|---|---|
| 散客弱贡献 | Alpha是受邀战队，散客场景有限；Beta开放后再加 |
| 半程战报 | 锦上添花，周战报已满足阶段性反馈需求 |
| 新手保护期 | Alpha受邀战队都是成熟队伍，不需要保护 |
| 即时战役 | 超出MVP范围 |
| 远征成就 | 超出MVP范围 |
| 经销商标识/联盟 | 超出MVP范围 |
| 骑行加成系数 | 复杂度高，先用纯里程验证 |
| 逐级统一成就 | 叙事增强，非核心闭环 |
| 周中随机事件 | 运营型功能，Alpha阶段先跑稳核心循环 |
| 社交分享H5 | Growth阶段获客工具 |
| 商业变现 | Vision阶段 |

### Risk Mitigation Strategy

**技术风险：**

| 风险 | 等级 | 缓解方案 |
|---|---|---|
| ECharts渲染2800县区性能 | ★★★★★ | 按缩放比例按需加载：全国概览→省级简化图→下钻到省再加载县区数据。Alpha前必须在低端机（iOS 13/Android 8）上验证 |
| 地图数据体积过大 | ★★★★ | 地图GeoJSON分省懒加载+CDN缓存+gzip压缩 |
| Mark独立全栈开发瓶颈 | ★★★★ | AI辅助提效；MVP砍到9项核心功能；Alpha不做运营后台 |
| 骑行数据对接稳定性 | ★★★ | 依赖顽鹿内部API，需提前确认接口文档和可用性 |
| 数值平衡（城防值/攻击力） | ★★★ | Alpha用固定公式，根据真实数据逐步调优 |

**市场风险：**

| 风险 | 缓解方案 |
|---|---|
| 空城综合征（活跃区域太少） | Alpha精选50个分布在不同地区的战队，确保地图不显得空旷 |
| 会长疲劳不操作 | 智能推荐+一键确认+不操作自动兜底三层保障 |
| 核心循环不够有趣 | Alpha快速迭代，2周一个小版本，4周一次回顾 |

**资源风险：**

| 风险 | 应对 |
|---|---|
| 开发周期超出预期 | 9项功能按优先级排序，最坏情况可再砍智能推荐简化为手动选择 |
| AI辅助效果不如预期 | 关键模块（攻防计算、地图渲染）提前做技术原型验证 |
| 一人扛不住运维 | Alpha阶段50个战队规模可控；监控+告警自动化 |

## Functional Requirements

> **能力契约**：本章定义产品的完整能力边界。UX设计师只设计、架构师只支撑、开发只实现此处列出的能力。未列出的能力不会出现在最终产品中。
>
> **术语说明**："战队"为本游戏中的玩家组织单位，独立于顽鹿运动App现有的"俱乐部"功能，不共用数据和接口。

### 地图与领土（Map & Territory）

- FR1: [Alpha] 用户可以查看全国领土概览地图，显示各区域的战队归属状态
- FR2: [Alpha] 用户可以从全国→省级→县区级逐级下钻浏览地图
- FR3: [Alpha] 用户可以查看任意县区的详细信息（归属战队、城防值、当前攻防状态）
- FR4: [Alpha] 用户可以在地图上识别无主之地、己方领土、敌方领土（通过视觉差异）
- FR5: [Alpha] 用户首次打开时可以看到自己家乡所在区域的领土状态

### 战队管理（Team Management）

- FR6: [Alpha] 用户可以创建一个新战队
- FR7: [Alpha] 战队在达到3名成员时自动激活为正式战队
- FR8: [Alpha] 用户可以搜索并申请加入已有战队
- FR9: [Alpha] 会长可以审批成员加入申请
- FR10: [Alpha] 战队可以选择一个县区作为大本营（初始领土）
- FR11: [Alpha] 用户可以查看自己战队的成员列表、领土列表、本周战况
- FR12: [Alpha] 成员可以主动退出当前战队
- FR13: [Alpha] 会长可以解散战队
- FR14: [Alpha] 成员退出或战队解散后，已结算的历史骑行贡献归属不变
- FR14a: [Alpha] 成员加入新战队后存在冷却期（≥72小时），冷却期内骑行里程不计入新战队攻防贡献（防止结算前批量转会刷榜）

### 骑行贡献（Ride Contribution）

- FR15: [Alpha] 系统可以获取用户在顽鹿运动App中完成的骑行数据（里程）
- FR15a: [Alpha] 系统对骑行数据执行基线校验：平均速度超过60km/h的骑行记录自动标记为异常并暂不计入；单次骑行时长低于5分钟的记录不计入；单用户单日累计里程超过300km的部分暂不计入，待运营复核
- FR16: [Alpha] 用户每次骑行完成后，里程自动转化为攻防贡献值
- FR17: [Alpha] 用户可以在骑行结束后立即看到本次骑行对攻城/守城进度的影响
- FR18: [Alpha] 用户可以查看自己的历史骑行贡献记录

### 宣战与攻防（War Declaration & Combat）

- FR19: [Alpha] 会长可以在宣战窗口期选择目标区域发起宣战
- FR20: [Alpha] 所有宣战在对战期开始时公开展示
- FR21: [Alpha] 对战期内，攻守双方成员的骑行里程实时累计为攻击力/防御力
- FR22: [Alpha] 用户可以查看当前参与的攻防战的实时进度
- FR23: [Alpha] 系统在结算期自动判定攻防胜负并更新领土归属
- FR24: [Alpha] 攻城成功时，领土归属变更立即反映在地图上
- FR24a: [Alpha] 系统在每周新对战期开始时对所有已占领区域施加城防衰减（城防值按固定比例下降），未被主动守护的领土逐渐变为易攻目标
- FR24b: [Alpha] 每个区域同一时间只能被一个战队占领（排他性占领模式）；多个战队可同时攻打同一无主区域，按周结算时累计攻击里程最高者获得占领权；攻打已占领区域时为攻守对抗模式

### 会长作战指挥（Captain Command）

- FR25: [Alpha] 会长可以收到系统智能推荐的作战目标
- FR26: [Alpha] 会长可以一键确认推荐的作战计划
- FR27: [Alpha] 会长可以手动修改作战目标，不采纳推荐
- FR28: [Alpha] 若会长在宣战窗口期内未操作，系统自动执行兜底方案

### 周战报（Weekly Report）

- FR29: [Alpha] 系统在周日12:00生成并发布本周战报
- FR30: [Alpha] 用户可以查看本周战报，包含领土变化、攻防结果、战队排名变化
- FR31: [Alpha] 用户可以查看自己在本周的个人贡献在战队中的排名

### 首次体验与引导（Onboarding）

- FR32: [Alpha] 新用户首次打开游戏时，系统自动定位到用户所在区域并展示家乡领土状态
- FR33: [Alpha] 新用户可以完成引导流程（包含≥3个步骤），引导流程覆盖核心玩法概念（骑行贡献、领土攻防、战队协作），完成率作为可度量验收标准
- FR34: [Alpha] 新用户可以在引导中选择创建战队或加入已有战队

### 系统周期管理（System Cycle）

- FR35: [Alpha] 系统按固定周循环运行：对战期→结算期→战报发布→宣战窗口→新对战期
- FR36: [Alpha] 系统在周循环各阶段切换时自动执行以下操作：对战期→结算期：锁定骑行数据并触发结算计算；结算期→战报发布：生成战报并更新领土归属；宣战窗口→新对战期：公布宣战声明并应用城防衰减
- FR37: [Alpha] 管理员可以查询当日活跃战队数、骑行里程总量、异常数据条目等关键运行指标，并可标记异常数据、暂停里程计入、修正结算结果（Alpha阶段通过预置运营工具实现）
- FR37a: [Alpha] 管理员可以导出指定时间范围的骑行贡献数据、战队状态数据、结算结果数据
- FR37b: [Alpha] 系统记录所有管理员操作（标记异常、修正结算、暂停计入等）并保留操作日志，包含操作人、操作时间、操作内容和影响范围
- FR37c: [Alpha] 系统在关键流程异常时自动触发告警（结算失败、数据同步队列积压超阈值、骑行数据服务不可用），告警通知到运营团队

### 推送通知（Push Notification）

- FR38: [Alpha] 系统可以在战队领土被宣战时向相关成员推送提醒
- FR39: [Alpha] 系统可以在周战报发布时向所有参与用户推送通知
- FR40: [Beta] 系统可以在攻城/守城进度接近临界点时向相关成员推送提醒

### 数据可见性（Data Visibility）

- FR41: [Alpha] 用户可以查看其他战队的成员列表和公开信息
- FR42: [Alpha] 用户可以查看其他战队成员的骑行贡献数据
- FR43: [Alpha] 用户可以查看战队排行榜，排行榜支持按总里程和人均贡献里程两个维度排序
- FR44: [Alpha] 用户可以在排行榜中按战队规模分组查看（如按活跃人数区间筛选），让不同规模的战队在各自群体中获得排名参照

## Non-Functional Requirements

### 性能（Performance）

| NFR | 指标 | 目标 | 测量方式 |
|---|---|---|---|
| NFR1 | 首屏加载时间 | P90 < 1秒（4G网络） | 真机测试，含地图框架初始化 |
| NFR2 | 地图交互帧率 | 缩放/拖拽时 ≥30fps | 低端机实测（iOS 13 / Android 8） |
| NFR3 | 地图下钻响应 | 点击省→加载县区数据 < 500ms | CDN + 缓存命中场景 |
| NFR4 | API响应时间 | P95 < 500ms（常规接口） | 后端监控 |
| NFR5 | 骑行即时反馈延迟 | 骑行结束→攻城进度更新可见 < 30秒 | 端到端链路测试 |
| NFR6 | 并发支持 | 500+同时在线用户（Alpha规模） | 压力测试 |

### 安全（Security）

| NFR | 要求 | 说明 |
|---|---|---|
| NFR7 | 用户认证复用顽鹿体系 | 用户认证通过宿主App传递凭证，后端每次请求验证凭证有效性 |
| NFR8 | 骑行数据来源校验 | 只接受顽鹿运动App上报的官方骑行数据，不接受外部导入 |
| NFR9 | 异常骑行检测 | 系统可以标记异常骑行记录（如凌晨3点超长里程），暂停计入并通知运营 |
| NFR10 | API限流 | 每用户每秒≤10次请求，防止恶意刷接口 |
| NFR11 | 数据传输加密 | 所有API通信使用HTTPS |

### 可扩展性（Scalability）

| NFR | 阶段 | 目标 |
|---|---|---|
| NFR12 | Alpha | 50个战队，~1500用户，500并发 |
| NFR13 | Beta | 200个战队，~6000用户，2000并发 |
| NFR14 | GA | 不限战队数，支撑15万MAU峰值 |
| NFR15 | 架构要求 | Alpha→Beta扩展不需要重写核心架构，只需水平扩容 |

### 可用性/可靠性（Reliability）

| NFR | 要求 | 说明 |
|---|---|---|
| NFR16 | 系统可用性 | Alpha阶段 ≥99%（每月停机<7.2小时） |
| NFR17 | 周结算零失败 | 结算流程必须完整执行或完整回滚，不允许部分结算 |
| NFR18 | 数据持久性 | 每日自动备份，备份可在1小时内恢复 |
| NFR19 | 错误恢复 | 系统异常重启后，自动恢复到最近一致状态，不丢失已提交的骑行贡献 |
| NFR20 | 结算幂等性 | 结算流程可重复执行，结果一致（防止重复计算导致数据错误） |

### 集成（Integration）

| NFR | 要求 | 说明 |
|---|---|---|
| NFR21 | 宿主App接口响应时间 | 宿主App原生接口调用响应时间 < 200ms |
| NFR22 | 骑行数据同步频率 | 用户完成骑行后，数据在30秒内可被游戏系统获取 |
| NFR23 | 推送到达率 | 依赖顽鹿推送服务，目标推送到达率 ≥ 90%，通过推送服务日志按日统计 |
| NFR24 | 集成降级 | 当骑行数据服务不可用时，游戏其他功能正常运行，里程贡献延迟处理；数据服务恢复后 ≤ 30分钟内完成积压里程补偿处理 |
